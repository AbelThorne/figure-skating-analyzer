# HTML-First Import Flow

## Problem

The current import flow downloads PDF score cards and tries to parse them for results. This is fragile — the PDF parser was extracting "JUDGES DETAILS PER SKATER" as the skater name, missing categories, clubs, and TES. Meanwhile, the competition website already has clean structured HTML tables with all this data.

## Decision

Use the competition website HTML as the primary data source for results. PDFs become optional enrichment for element-by-element details.

## Site Structure (FS Manager / Swiss Timing)

Competition sites generated by FS Manager follow a consistent layout:

```
index.htm
├── Event row 1: "R1 Junior-Senior Femme"
│   ├── CATxxxEN.htm  — entries / starting list
│   ├── CATxxxRS.htm  — overall results (rank, name, club, nation, total points)
│   └── SEGxxx.htm    — segment details (TSS, TES, PCS, CO, PR, SK, Ded, StN)  ← richest source
│       └── PDF link   — judges details per skater (element-by-element breakdown)
├── Event row 2: "R1 Junior-Senior Homme"
│   └── ...
└── ...
```

The **SEG pages** are the primary import target. They contain: placement, name, club, nation, total segment score, total element score, program component score, individual components (CO/PR/SK, plus IN/CH for higher levels), deductions, and starting number.

## Two-Step Flow

### Step 1: Import (HTML scraping)

`POST /api/competitions/{id}/import`

1. Fetch `index.htm`, parse the event table
2. For each event, extract the SEG page URL and category/segment info
3. Fetch each SEG page, parse the results table
4. For each competitor row:
   - Get or create `Skater` by name (enrich with club, nationality)
   - Create `Score` with: category, segment, rank, total_score, technical_score, component_score, components (JSON), deductions, starting_number
5. Return: `{events_found, scores_imported, errors}`

Import is **idempotent** — re-importing skips scores that already exist. Add a unique constraint on `(competition_id, skater_id, category, segment)` to the Score model. On conflict, skip (don't update).

### Step 2: Enrich (PDF parsing) — optional

`POST /api/competitions/{id}/enrich`

1. Fetch `index.htm`, collect PDF URLs from event rows
2. Download PDFs to local storage
3. Parse each PDF for element-by-element details
4. Match to existing scores by (skater name + category + segment)
5. Update the `elements` JSON field on matched scores
6. Return: `{pdfs_downloaded, scores_enriched, unmatched, errors}`

## Data Model Changes

### Score model — add two JSON fields

```python
# Individual component marks from SEG page (e.g. {"CO": 3.42, "PR": 3.25, "SK": 3.25})
components: Mapped[Optional[dict]] = mapped_column(JSON, nullable=True)

# Element-by-element details from PDF enrichment
# e.g. [{"name": "3Lz", "base_value": 5.90, "goe": 0.59, "score": 6.49}, ...]
elements: Mapped[Optional[dict]] = mapped_column(JSON, nullable=True)
```

### Database migration

The project uses `create_all` (no Alembic). Since this is early development with no production data, **delete the existing `skating.db`** before restarting. The `init_db()` call will recreate all tables with the new columns.

### Existing columns

- `pdf_path` — retained, set during enrichment only (null for HTML-imported scores)
- `raw_data` — retained for debugging/diagnostics, stores raw page text or parser output

### Skater matching

Match by name for now. The `Skater` model keeps `unique=True` on `name`. This is a known limitation — name collisions across different clubs are rare in regional French competitions. If needed later, switch to compound key (name + club).

## Site Scraper Rewrite

Replace `FrenchIJSScraper` with `FSManagerScraper`.

### Index page parsing

The index table has paired rows per event:

```html
<!-- Row 1: category + entries/results links -->
<tr class="Line1Red">
    <td class="CellLeft">R1 Junior-Senior Femme</td>
    <td></td>
    <td class="CellLeft"><a href="CAT001EN.htm">Entries</a></td>
    <td class="CellLeft"><a href="CAT001RS.htm">Result</a></td>
    <td>&nbsp;</td>
</tr>
<!-- Row 2: segment + panel/classification/PDF links -->
<tr class="Line1Red">
    <td class="CellLeft" valign="top"></td>
    <td class="CellLeft">Free Skating</td>
    <td class="CellLeft"><a href="SEG018OF.htm">Panel of Judges</a></td>
    <td class="CellLeft"><a href="SEG018.htm">Starting Order / Detailed Classification</a></td>
    <td><a href="...pdf">Judges Scores (pdf)</a></td>
</tr>
```

Detection: row 1 has a non-empty first cell (category name), row 2 has an empty first cell and segment name in the second cell. A category may have multiple segment rows (e.g. SP + FS) — collect all of them under the same category.

### SEG page parsing

Table headers: `Pl. | Name | Club | Nation | TSS= | TES+ | (spacer) | PCS+ | CO | PR | SK | Ded.- | StN.`

Data rows contain: rank (int), name (link text), club, nation (text in nested flag table), TSS (float), TES (float), spacer, PCS (float), component values (floats), deductions (float), starting number (e.g. "#8").

The nation column contains a nested `<table>` with a flag image and text — extract the text node with the 3-letter code.

### Category/segment extraction

Category and segment are already known from the index page parsing (row 1 = category, row 2 = segment). No need to parse them from the SEG page title.

Map segment names: "Free Skating" → "FS", "Short Program" → "SP", "Rhythm Dance" → "RD", "Free Dance" → "FD".

## PDF Element Parsing (Enrichment)

The PDF "Judges Details per Skater" contains per-skater blocks with a table of executed elements:

```
# Executed Elements  Base  Scores of  GOE  J1 J2 J3  Ref.
                     Value Panel
1 3Lz               5.90  6.49       0.59  1  1  1
2 2A                3.30  3.63       0.33  1  1  1
```

The parser extracts from each skater block:
- Skater name (from the header line: "1 Gaia-Lior GARNIER-RAYAR FRA 4 6.69 ...")
- List of elements: `[{"number": 1, "name": "3Lz", "base_value": 5.90, "score": 6.49, "goe": 0.59}, ...]`

Matching to DB scores: use the skater name + category (from the PDF header line, e.g. "R3 C BABIES FEMME FREE SKATING") mapped to the stored category + segment.

## Error Handling

- Partial success: if some SEG pages fail to fetch, import the rest and report errors in the response
- Each event is processed independently — one failure doesn't block others
- Transaction: commit all successful scores at the end, report failed events in `errors` array

## Not In Scope

- Frontend changes (enrich button) — separate task
- Handling non-FS-Manager sites — `GenericTableScraper` stays as-is
- Entries page parsing (`CATxxxEN.htm`) — not needed, SEG pages have all the data
